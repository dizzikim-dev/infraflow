

📌 Meta 스태프 엔지니어가 Claude Code를 사용하는 핵심 팁은 무엇인가요?
Claude Code를 효과적으로 사용하기 위한 50가지 팁으로, 루트 디렉토리에서 실행, /init으로 초기 설정, CLAUDE.md 파일 관리, 키보드 단축키, 필수 명령어, Git 활용, 병렬 개발 워크플로우 등이 핵심입니다.









메타 스태프 엔지니어가 6개월간 매일 사용하며 터득한 AI 코딩 생산성 극대화 비법 50가지를 공개합니다. 초보자가 흔히 저지르는 맥락(Context) 낭비 함정을 피하고, '계획 모드'와 '검증 루프'를 활용해 AI의 결과물을 즉시 신뢰하는 대신 스스로 개선하게 만드는 실전 전략을 배울 수 있습니다. 이 팁들을 익히면 AI와 협업하는 방식이 완전히 달라지며, 코딩 속도와 품질을 동시에 높일 수 있습니다.



1. Claude Code 사용 기초 및 설정
Claude Code 사용을 위한 초기 설정, 기본 명령어, 그리고 핵심 설정 파일인 claw.md의 기본 구조를 이해한다.



1.1. 초기 설정 및 기본 실행
captureSource
프로젝트 루트 디렉토리에서 실행 

Claude 명령어를 사용하여 현재 작업 중인 프로젝트의 루트 디렉토리에서 Claude Code를 실행해야 한다. 

이는 초기 설정 파일(claw.md)이나 기타 설정이 있을 경우, Claude가 첫 번째 토큰으로 컨텍스트를 압축할 때 루트 디렉토리를 기준으로 하기 때문이다. 



/init 명령어 실행 

Claude Code를 실행한 후, /init 명령어를 실행한다. 

이 명령어는 코드베이스를 분석하여 초기 claw.md 파일을 프로젝트 내 cloud 디렉토리에 생성한다. 

분석 결과(예: Next.js 15 앱, 포트폴리오 사이트 등)를 확인하고 Yes를 눌러 커밋하는 것이 중요하다. 



1.2. claw.md 파일의 계층 구조 및 관리
captureSource
claw.md의 계층 구조 

claw.md 파일은 계층 구조(Hierarchy)로 작동한다. 

체크인된 메모리(Checked-in memory)는 현재 루트 디렉토리를 의미하며, 사용자 메모리(User memory)는 전역 설정(예: ~/.claude/claude)을 의미한다. 

이 구조를 통해 Claude Code가 현재 어떤 핵심 규칙(Core Rule)으로 작동하고 있는지 알 수 있다. 



claw.md의 적절한 크기와 내용 구성 

claw.md 파일은 너무 크지 않게 유지하는 것이 좋으며, 약 300줄 정도가 적절하다. 

컨텍스트 창이 커질수록 토큰 사용량이 증가하며, 컨텍스트에 불필요한 내용(Bloat)이 많을수록 AI가 원하는 대로 작동할 가능성이 낮아진다. 

포함해야 할 주요 내용은 다음과 같다. 

고수준 기술 아키텍처 및 요구사항 

도메인 컨텍스트 (예: SwiftUI 사용, 디자인 패턴, 파일 경로 등) 

빌드 및 검증 흐름(Validation Flow) 

검증 루프(Validation Loop)를 포함하는 것이 매우 중요하며, 이는 AI가 스스로 개선하고 문제를 해결하도록 돕는다. 

AI 코딩 시스템에서 검증 루프는 AI의 성능을 극적으로 향상시키는 가장 중요한 요소 중 하나이다. 



1.3. 필수 키보드 단축키
captureSource
모드 전환: Shift + Tab 

Shift + Tab을 눌러 수락 모드(Accept Edits)와 계획 모드(Plan Mode)를 전환할 수 있다. 

계획 모드를 새 기능 시작 시 거의 독점적으로 사용하며, 코드베이스에 대한 가정을 검증하는 데 시간을 투자한다. 



실행 중단 및 입력 관리 

Escape 키는 현재 진행 중인 작업을 중단(Interrupt)시킨다. 

계획 모드 중 AI의 생각이 궤도를 벗어난다고 판단되면 즉시 Escape를 눌러 중단하고 방향을 수정하는 것이 권장된다. 

두 번의 Escape를 누르면 입력창의 내용이 지워진다(Clear Input). 

빈 입력창에서 두 번의 Escape를 누르면 이전 시점으로 되돌아가 컨텍스트를 복원할 수 있다. 



컨텍스트 관리 및 기타 단축키 

Claude Code의 많은 기능은 컨텍스트 관리 및 컨텍스트 엔지니어링을 위해 설계되었다. 

스크린샷을 찍어 드래그 앤 드롭으로 컨텍스트에 추가할 수 있으며, UI 작업 시 유용하다. 

Figma MCP 등 특정 검증에 맞는 MCP(Model Context Protocol)를 찾는 것이 좋다. 

더 많은 단축키는 /help 명령어를 통해 확인할 수 있다. 



2. 필수 슬래시 명령어 (Essential Commands)
컨텍스트 관리, 모델 선택, 세션 복구 등 핵심적인 작업을 수행하는 데 필요한 슬래시 명령어를 익힌다.



2.1. 컨텍스트 및 세션 관리 명령어
captureSource
/clear: 현재 컨텍스트를 지운다. 

새 기능을 시작할 때 이전 작업의 컨텍스트가 새 프로젝트에 영향을 주는 것을 방지하기 위해 사용한다. 



/context: 현재 컨텍스트의 시각적 표현을 제공한다. 

Claude Code가 코드베이스에 대해 내리는 가정(Assumptions)을 검토할 기회를 제공한다. 

컨텍스트가 신선하고 압축되어 있을 때(Fresh and Condensed) 가장 좋으며, 성능 저하 또는 비용 증가 시 감사(Audit)해야 한다. 

MCP는 토큰 사용량을 급증시키는 주범 중 하나이므로, /context를 통해 확인 후 제거하거나 비활성화할 수 있다. 



/compact: 현재 컨텍스트를 요약하여 압축한다. 

긴 세션 후에는 자동 압축(Auto-compaction)이 작동하지만, 수동으로 압축하여 로컬 '세컨드 브레인'에 저장할 때 유용하다. 



/model: 사용할 모델을 변경한다. 

토큰 제한이 없다면 Opus 4.5를 기본으로 사용하는 것이 좋으나, 비용에 민감하다면 Sonnet 등으로 변경할 수 있다. 



/resume: 세션을 복구한다. 

실수로 인스턴스를 종료했을 때 이전 컨텍스트를 복구하여 작업 손실을 막는다. 



2.2. MCP 및 도움말 명령어
captureSource
/mcp: 설치된 MCP(Model Context Protocol)를 보여준다. 

MCP는 컨텍스트 창을 비대하게 만들 수 있으므로, 꼭 필요한 경우에만 설치하고 수동 스크립트 작성을 선호하는 것이 좋다. 

MCP는 타사 소프트웨어 교육 등에 유용하며, 토큰 사용량 증가에 주의해야 한다. 



/help: 모든 명령어 목록을 제공한다. 

사용자가 직접 탐색하고 시도해 볼 수 있도록 돕는 가이드 역할을 한다. 



2.3. Git 및 claw.md 심층 분석 준비
captureSource
Git 활용의 중요성 

Claude Code에게 Git 관리를 맡길 수 있으며, 요약이나 테스트 계획 작성에 활용 가능하다. 

Git을 안전망(Safety Net)으로 사용하는 것을 강력히 권장하며, Claude의 자체 되감기 기능은 Git보다 덜 명확하여 최후의 수단으로 사용한다. 

이제 claw.md 파일에 대해 심층적으로 다룰 차례이다. 



3. claw.md 심층 분석 및 고급 규칙
규칙 파일인 claw.md를 효과적으로 작성하고 관리하는 방법과, 위험한 작업을 방지하는 고급 설정을 다룬다.



3.1. claw.md 작성 원칙
captureSource
우선순위 및 내용 구성 

claw.md는 위에서 아래로(Top to Bottom) 읽히며, 위쪽 내용이 더 높은 우선순위를 갖는다. 

"절대 ~하지 말라"는 규칙과 "항상 ~하라"는 규칙을 명시하고, 명확한 코드 스니펫 예시를 포함하는 것이 좋다. 



고유한 프로젝트 패턴 정의 

프로젝트의 고유성(Homegrown)이 높다면, AI가 학습하지 못한 특수한 DSL 언어나 오래된 패턴에 대해 규칙을 명시해야 한다. 

이는 린트 파일(Lint files)과 유사하게 작동하며, AI의 실수를 방지한다. 



규칙 업데이트 자동화 

규칙을 수동으로 수정하지 말고, Claude에게 "저 규칙을 업데이트해 달라"고 요청하여 관리해야 한다. 



3.2. 워크플로우 트리거 및 컴파운드 엔지니어링
captureSource
키워드를 이용한 스킬 트리거 

규칙 파일 내에 빌드 명령어와 같은 트리거 단어를 정의하여 특정 행동(스킬)을 유발할 수 있다. 



컴파운드 엔지니어링 (Compound Engineering) 

claw.md를 코드베이스에 커밋하는 개념이다. 

커밋 시 일반적인 내용(파일 경로 등)은 제거해야 하며, 파일 크기가 커지지 않도록 주의해야 한다. 

이 방식은 팀원들의 AI 코딩 경험을 향상시키지만, 실제로 도움이 되는지에 대한 평가가 필요하다. 



3.3. 위험 회피 설정: dangerously-skip
captureSource
--dangerously-skip 옵션 

Claude Code가 실행 전 수락을 요청하는 과정을 건너뛰고 싶을 때 사용한다 (YOLO 모드). 

이 옵션은 폐기 가능한 환경(Throwaway environments)에서 위험하지 않은 작업을 할 때만 사용해야 한다. 

/permissions 슬래시 명령어를 사용하여 rm -rf와 같은 파괴적인 명령어에 대해서는 여전히 수락을 요구하도록 설정할 수 있다. 



4. 일일 작업 흐름 및 컨텍스트 전략
저자가 실제로 사용하는 Claude Code 기반의 개발 워크플로우와, 컨텍스트를 효율적으로 구축하고 유지하는 전략을 설명한다.



4.1. 계획 모드 중심의 개발 워크플로우
captureSource
계획 모드 우선 사용 

항상 계획 모드로 시작하여 Claude Code와 반복적으로 대화하고 논쟁하며 계획을 다듬는다. 

AI를 동료 엔지니어처럼 대하며, 첫 번째 답변을 순수하게 수락하지 않고 검증한다. 

이 단계에서 충분한 시간과 노력을 투자하여 좋은 실행 사양(Execution Specs)을 가진 컨텍스트를 구축하면, 코드 생성 자체는 쉬워진다. 



코딩의 미래와 추상화 수준 

수동으로 코드를 작성하는 시대는 끝났다는 의견에 동의하며, 현재 작업 방식은 새로운 추상화 수준이다. 

엔지니어링의 다음 단계는 여러 Claude 인스턴스를 오가며 컨텍스트 엔지니어링을 수행하는 것이다. 



신선한 컨텍스트의 중요성 

신선한 컨텍스트(Fresh Context)가 부푼 컨텍스트(Bloated Context)보다 우세하다. 

여러 시도를 반복하는 대신, 초기에 시간을 들여 좋은 계획과 컨텍스트를 구축해야 AI가 혼란스러워지는 것을 방지할 수 있다. 



4.2. 컨텍스트 지속 및 관리 전략
captureSource
컨텍스트 지속(Persisting Context) 

작업 완료 시, 로컬 claw.md에 해당 작업을 저장하도록 지시하여 '세컨드 브레인' 개념을 활용한다. 

이 세컨드 브레인에서 컨텍스트를 로드하여 새 세션에서 재사용할 수 있다. 



To-Do 목록 관리 및 지연 로딩(Lazy Loading) 

To-Do 목록을 Claude Code 내에 기록하고, 필요할 때만 로컬 인덱스에서 지연 로딩한다. 

이를 통해 복잡한 프로젝트 관리 도구 없이도 프로젝트 간 컨텍스트 전환(Context Switch)이 용이해진다. 



4.3. 검증 루프의 확장 및 모델 선택
captureSource
다양한 검증 루프 구상 

모바일 개발 시 Xcode 빌드 외에도 디버깅 시 디버그 로그 추가 및 에뮬레이터 제어를 요청할 수 있다. 

성능 검증을 위해 Peretto MCP를 활용하여 트레이스를 읽고 잰크(Jank)를 찾을 수 있다. 

웹 개발에서는 Puppeteer를 사용하여 크롬을 제어하고 탐색하거나 E2E 테스트를 작성할 수 있다. 

검증 루프를 구상하는 것이 첫 시도에 좋은 코드를 생성하는 데 가장 중요하다. 



모델 선택 및 병렬 개발 

가능하다면 Opus 모델을 사용하는 것이 최선이다. 

저자는 여러 Claude 인스턴스를 동시에 실행하며 컨텍스트를 구축하는 방식으로 작업 속도를 유지한다. 



중단 및 방향 수정 

AI가 가정(Assumptions)을 하거나 오류가 발생하면 즉시 중단하고 방향을 수정(Course Correct)해야 한다. 



5. 파워 유저를 위한 구성 요소 (Primitives)
Claude Code의 핵심 구성 요소인 스킬, MCP, 서브 에이전트 등의 개념과 이들의 조합(컴포저빌리티)을 이해한다.



5.1. 컴포저빌리티 프레임워크의 네 가지 원시 요소
captureSource
Claude Code의 핵심은 이러한 요소들이 조합 가능(Composable)하다는 점이다. 

네 가지 주요 구성 요소(Primitives)가 있다. 



5.2. 스킬 (Skills)
captureSource
스킬의 정의 

스킬은 반복적인 워크플로우를 저장한 것으로, 두 가지 이상의 작업을 순서대로 수행하도록 정의된다. 

예: 해커 뉴스에서 최신 iOS 뉴스를 가져와 로컬 claw.md에 요약 저장. 



스킬 생성 및 저장 

위에서 수행한 워크플로우를 새로운 스킬(예: Fetch Hacker News)로 저장하면 된다. 

스킬은 내부적으로 시스템 프롬프트(System Prompt)로 저장되는 MD 파일이다. 

Claude는 스킬이 저장된 특정 디렉토리를 인식하여 다음 호출 시 추론하여 사용한다. 



명령어(Commands)와의 관계 변화 

최근 변경으로 스킬과 슬래시 명령어가 상호 교환 가능해졌다. 

이제 /fetch Hacker News와 같이 슬래시 명령어로 스킬을 바로 실행할 수 있으며, Claude가 추가적인 추론을 통해 동작을 확장하기도 한다. 



스킬 관리 

스킬은 수동으로 만들지 말고, Claude에게 업데이트 및 관리를 요청해야 한다. 

예: Fetch Hacker News 스킬을 Twitter나 Apple News까지 확장하도록 요청할 수 있다. 



5.3. MCP (Model Context Protocol)
captureSource
MCP 검색 및 설치 

MCP는 모델 컨텍스트를 제공하는 역할을 하며, 웹 검색 없이도 Claude에게 "좋은 Figma MCP를 찾아달라"고 요청할 수 있다. 

다만, 최신 버전이 아닐 수 있으므로 AI의 생각(Thinking) 과정을 확인하고 필요하면 중단해야 한다. 

Claude에게 MCP 설치를 요청할 수 있어 설정 관리가 용이하다. 



5.4. 서브 에이전트 (Sub-Agents)
captureSource
서브 에이전트 생성 및 목적 

서브 에이전트는 슬래시 명령어 및 스킬처럼 작업 수행 후 생성을 요청할 수 있다. 

주된 목적은 병렬 작업 수행과 컨텍스트 창 보호이다. 



서브 에이전트 사용 시 주의점 

서브 에이전트가 작업을 마치고 결과를 가져올 때, 결과만 가져올 뿐 그 과정(Full Context)은 포함하지 않는다. 

따라서 코드 작성이나 복잡한 테스트처럼 전체 컨텍스트가 필요한 작업은 메인 세션 내에 두는 것이 좋다. 

서브 에이전트는 원자적(Atomic) 성격이거나 부수 효과(Side Effects)로 실행되기를 원하는 작업에 가장 적합하다. 

CEO 에이전트, 제품 에이전트 등 컨텍스트를 분산시키는 방식은 지양해야 하며, 작업을 컨텍스트로 가져와야 한다. 



5.5. 컨텍스트 응집력 유지
captureSource
컨텍스트 응집력 유지 

지침(Instructions)을 과도하게 추가하거나 컨텍스트를 항상 오버로드하는 것을 피해야 한다. 

컨텍스트는 압축되고 신선하게 유지되어야 하며, '세컨드 브레인'을 RAG 시스템처럼 활용하여 좋은 컨텍스트를 지속적으로 제공해야 한다. 



6. 고급 병렬 개발 워크플로우 및 자동화
Claude Code의 가장 큰 장점인 병렬 개발 능력과, 이를 극대화하기 위한 터미널 활용, 자동화 훅(Hook) 설정을 다룬다.



6.1. 병렬 개발 및 터미널 활용
captureSource
다중 인스턴스 병렬 개발 

Claude Code의 가장 큰 혁신은 병렬 개발이 가능하다는 점이다. 

여러 인스턴스를 실행하고 전환하며 동시에 여러 작업을 진행할 수 있다 (예: 한쪽에서 라이브 액티비티 기능 작업, 다른 쪽에서 서브 에이전트 스폰). 



iTerm 활용 팁 

iTerm을 사용하면 Command + D로 새 인스턴스를 분할하고, Command + [, Command + ]로 터미널 간 전환이 용이하다. 

탭을 사용할 경우, 각 탭의 이름을 변경하여 현재 작업(예: Local, Remote SSH)을 명시하면 관리가 쉽다. 



입력 방식 및 알림 

Whisper와 같은 음성 변환 도구를 사용하여 타이핑 없이 대화하듯 작업할 수 있다. 

현재 병목은 머릿속에서 처리할 수 있는 컨텍스트 전환 횟수이다. 

알림 활성화를 통해 작업 완료 시 소리 알림을 받거나, 과거에는 텍스트 음성 변환(TTS) 요약 기능을 사용하기도 했다. 



Git Worktrees 활용 

여러 인스턴스로 로컬 개발을 할 때, Git Worktrees를 사용해야 동일 프로젝트의 코드 편집이 가능하다. 

Worktrees는 코드베이스의 여러 복제본을 만들어 포크된 브랜치처럼 작업할 수 있게 한다. 



6.2. /chrome 명령어와 웹 탐색
captureSource
/chrome 명령어 사용 

/chrome 명령어를 통해 브라우저를 열고 웹을 탐색할 수 있다. 



웹 탐색 및 자동화 

Claude가 계획을 세우고 사용자가 승인하면, AI가 웹사이트를 탐색하고 클릭하며 작업을 수행한다. 

API 접근이 불가능한 경우, 웹을 통해 접근하여 데이터를 가져오는 데 유용하다 (예: YouTube에서 요리 영상 찾기). 

이 기능은 검증 루프의 일부로 활용될 수 있으며, 폼 작성 등에도 사용 가능하다. 



6.3. 훅(Hooks) 및 자동화
captureSource
훅(Hooks)의 역할 

훅은 GitHub의 pre/post-commit 훅과 유사하게, 실행 전(Pre-hook) 또는 실행 후(Post-hook)에 특정 작업을 수행하도록 설정한다. 

이러한 설정은 수동이 아닌 Claude에게 요청하여 환경을 유지해야 한다. 



훅 활용 예시 

Post-tool use 훅에는 코드 린팅 및 포맷팅을 설정할 수 있다. 

데이터베이스 전체 삭제와 같은 파괴적인 작업을 방지하기 위해 훅을 설정할 수 있다. 



6.4. 플러그인 생태계 및 최종 결론
captureSource
플러그인 생태계 탐색 

플러그인은 스킬, MCP, 서브 에이전트, Bash 스크립트 등 조합 가능한 요소들의 결합체이다. 

온라인에서 공유되는 플러그인을 다운로드하여 활용할 수 있다. 



최종 메시지: 컨텍스트가 왕이다 (Context is King) 

컨텍스트는 신선하고 관련성 있게 유지해야 하며, Claude에게 필요한 컨텍스트만 제공하고 그 이상은 제공하지 않아야 한다.
