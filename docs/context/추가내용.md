영역 1) 인라인 참조 박스 (Reference Box)
접근 A 추천은 동의. 다만 “데이터 계약”을 분리해두면 더 안전해요

지금 설계처럼 ParseResultInfo(traceSummary + usedSources[])로 가는 게 최소 변경인데, 여기서 핵심은:

usedSources를 “raw list”로 그냥 내려주기보다
Answer-level aggregator로 한번 정제해서 내려주는 걸 추천해요.

dedupe(같은 URL 반복)

rank(official/vendor/community)

limit(기본 3~7)

왜? UI가 아니라 “응답 품질”이 소스 폭발에 흔들리지 않게 하려면, 정제 레이어를 백엔드(혹은 공용 util)에 두는 게 유지보수에 좋아요.

트레이드오프(“PromptPanel이 커짐”) 완화 팁

기본은 “접힘 + 한 줄 요약”만:
▸ 참조 출처 (5) · 검증 PASS · 경고 1

열었을 때만 상세 리스트

“자세히 보기”는 EvidencePanel로 넘기기(깊이는 사이드패널이 담당)

추가하면 신뢰가 확 오르는 2가지

검증 배지 연결: PostVerifier 결과를 ReferenceBox 헤더에 배지로 같이 노출

출처 사용 위치: usedInSteps: ["rag","verify"] 같은 태그만 있어도 “이 소스가 왜 여기 있나”가 설명됨

영역 2) 프로젝트 프로파일 (Project Profile)
접근 A(LocalStorage + Context Provider) 추천도 동의. 다만 “프로젝트 단위”가 핵심

LocalStorage로 시작하되, **“프로파일 1개”가 아니라 “프로젝트별 프로파일 N개”**가 되어야 진짜 쓸모가 커져요.

예: profiles: [{id, name, industry, sla, ...}]

PromptPanel 상단에 현재 선택된 profile 표시

RequirementsWizardPanel 완료 시: “이걸 프로파일로 저장” 버튼

트레이드오프(디바이스 간 공유 불가) 완화책

B(Prisma DB)로 가기 전에, A 단계에서 바로 가능한 가성비 기능:

Export / Import(JSON) 버튼

개인 사용자도 “노트/문서”처럼 옮길 수 있음

나중에 DB 저장으로 옮길 때도 마이그레이션이 쉬움

프로파일에서 꼭 넣어야 하는 운영 포인트 2개

schemaVersion: 필드 바뀌면 마이그레이션 필요

민감정보 금지 가이드: (예산 정도는 OK) 하지만 “API Key / 비밀번호 / 고객 실명 데이터” 같은 건 저장 금지 UX 문구 필요

영역 3) 모범답변 구조화 (Structured Response Template)
접근 A(1-pass + JSON 스키마)로 가되, “파싱 실패/구조 이탈”을 제품이 흡수해야 해요

이 영역의 진짜 트레이드오프는 “LLM이 안 지키면 끝”인데, 이걸 아래로 해결하면 안정화됩니다.

추천 패턴: 1-pass 기본 + 자동 복구(Repair) 옵션

1-pass로 structuredResponse 받기

파싱 실패 / 필수 필드 누락이면:

(추가 호출 없이) fallback 렌더링: “요약/전제/L1/L2/L3 중 가능한 것만”

또는 (필요 시에만) Repair pass(저렴한 모델/짧은 프롬프트)로 재구성

즉, 항상 2-pass가 아니라, “깨졌을 때만” 2-pass를 쓰는 방식.

스키마 설계 팁(진짜 중요한 것)

답변 본문 텍스트보다, UI가 필요한 메타를 스키마로 강제하는 게 효과가 큼

assumptions[]

options[] (최소 2개)

tradeoffs[]

artifacts[]

traceSummary + answerEvidence(sources[])

이렇게 하면 “서술”은 모델이 유연하게 쓰고, “구조/품질”은 시스템이 잡아요.

영역 4) 전제·확인 섹션 (Assumption Checker)
접근 A(규칙/온톨로지 기반) 추천도 동의. 다만 “질문을 늘리는 UX”가 되면 역효과가 나요

Assumption Checker는 잘못하면 “물어보는 봇”이 되거든요.

그래서 UX 규칙을 고정하는 게 중요

한 번에 최대 3~5개만 물어보기

“프로파일에 있으면 묻지 않기”

“필수/선택”을 구분해서, 필수만 우선

룰 정의 방식(가성비 좋게)

노드 타입/카테고리별 requiredAssumptions 테이블

예: DB → {rpo, rto, writeRatio, pii, encryptionAtRest}

WAF → {trafficRegion, compliance, threatModel}

지금 있는 PostVerifier의 missing 개념과 연결하면 더 자연스럽게 동작해요
(이미 검증이 “뭐가 부족한지”를 알고 있으니까)

접근 B(LLM 기반) 추가는 “백업 플랜”으로만

규칙이 커버 못하는 신규 도메인에서만

또는 사용자가 “이거 왜 또 물어봐?” 할 때, 이유 설명용으로

영역 5) 페르소나 기반 응답 적응 (Persona Adaptation)
접근 A(프로파일 + 프롬프트 분기) 좋습니다. 단, “콘텐츠의 사실/구조”는 바꾸면 안 돼요

페르소나는 정답을 바꾸는 게 아니라 ‘강조/깊이/용어 수준’을 바꾸는 것이 핵심이에요.

추천 구현: persona를 3개의 축으로 분해(고정 페르소나의 단점 완화)

Depth: 요약 ↔ 디테일

Focus: 비용/일정 ↔ 보안/증빙 ↔ 운영/SRE ↔ 개념학습

Tone: 컨설팅 톤 ↔ 학습 톤

UI는 “저는 ___입니다” 셀렉터 그대로 두되, 내부적으로는 위 3축을 세팅하는 방식이면

페르소나 추가/변형이 쉬워지고

프롬프트도 덜 지저분해져요.

(추가 제안) 5개에 “자연스럽게 붙는” 2개 기능

이미 다른 메시지에서 언급된 것과도 맞물려요.

Cost 민감도 분석(3안: 상한/기준/하한)

E(비용) 질문에 체감이 큼

“입력 부족”을 시나리오로 흡수 가능 → Assumption Checker의 부담도 줄어듦

입력 단계 Secret/PII 탐지

OUTPUT 탐지는 있는데 INPUT이 없다고 했죠.

프로파일 저장/내보내기 기능이 생기면 INPUT 탐지가 더 중요해져요(리스크가 커짐)
